# Style Reference Ingestion

## 목적
문체를 “감으로 맞추는 일”이 아니라, **분석 가능한 규칙**으로 바꾸는 것.
좋은 스타일 적용은 “누구 글을 흉내 냈다”가 아니라, “이 사용자의 톤과 리듬이 살아 있으면서도 목적에 더 맞다”에 가깝다.

## 1. 어떤 입력을 받을 수 있나
- 사용자가 직접 쓴 글 2~5개
- 브랜드/조직 톤앤매너 문서
- 기존 뉴스레터/칼럼/블로그 샘플
- 메모, 스레드, 짧은 포스트 묶음
- 참고하고 싶은 타인 글
- 금지 표현/선호 표현 목록

## 2. 우선순위
1. 사용자가 직접 쓴 글
2. 사용자가 만든 문체 가이드 파일
3. 조직/브랜드 문서
4. 타인 레퍼런스
5. 형용사 수준의 취향 설명

## 3. 분석 원칙
문체는 아래 10가지로 나눠 본다.

1. **목소리**: 차분한가, 친근한가, 단호한가
2. **관점 거리**: 1인칭 중심인가, 설명자형인가
3. **문장 길이**: 짧고 리듬감 있는가, 길고 사유형인가
4. **문단 호흡**: 짧은 단락 위주인가, 긴 단락 위주인가
5. **어휘 수준**: 쉬운 단어 중심인가, 개념어가 많은가
6. **전환 습관**: 그런데/다만/즉/결국 같은 연결 방식
7. **강조 방식**: 질문, 대조, 반복, 짧은 결론 문장
8. **사례 사용법**: 경험, 비유, 데이터, 인용 중 무엇을 선호하는가
9. **오프닝/엔딩 습관**: 장면, 명제, 질문, 요약 중 무엇으로 시작/끝내는가
10. **말버릇/금지어**: 자주 쓰는 표현과 피해야 할 표현

## 4. Style Card 만들기
샘플이 충분하면 내부적으로 아래 형식의 `Style Card`를 만든다.

### Style Card 정규 형식

```
# Style Card

- profile_name:
- source_samples:
- default_intensity: medium
- updated_at:

## 1. voice
- keep:
- avoid:

## 2. perspective_distance
- keep:
- avoid:

## 3. sentence_length
- keep:
- avoid:

## 4. paragraph_rhythm
- keep:
- avoid:

## 5. vocabulary_level
- keep:
- avoid:

## 6. transition_habits
- keep:
- avoid:

## 7. emphasis_patterns
- keep:
- avoid:

## 8. evidence_usage
- keep:
- avoid:

## 9. opening_ending_patterns
- keep:
- avoid:

## 10. verbal_habits_banned_phrases
- keep:
- avoid:

## Change Log
- YYYY-MM-DD: keep/discard | dimension | reason | note
```

이 형식은 Claude Code 스킬과 동일합니다. GPTs에서 내보낸 Style Card를 Claude Code에서 그대로 사용할 수 있고, 그 반대도 동일합니다.

## 5. 문체 적용 강도
사용자에게 필요하면 아래 강도를 제시한다.

- **약**: 문체 느낌만 반영, 목적과 가독성 최우선
- **중**: 단어 선택, 문장 호흡, 전환 습관까지 반영
- **강**: 구조적 버릇, 리듬, 서술 거리까지 적극 반영

기본값은 **중**이다.

## 6. 사용자의 글 vs 타인 레퍼런스
### 사용자의 자기 문체
가능한 한 가깝게 반영해도 된다.

### 타인 레퍼런스
문장을 흉내 내지 말고 아래만 추출한다.
- 톤
- 밀도
- 설명 방식
- 구조
- 리듬
- 사례 처리

즉, “복제”보다 “고수준 특성 이전”을 한다.

## 7. 샘플이 빈약할 때
아래 경우는 과적합을 피한다.
- 샘플이 1개뿐
- 길이가 매우 짧음
- 장르가 서로 크게 다름
- 사용자의 실제 글보다 타인 레퍼런스가 많음

이때는 문체 확신도를 낮게 보고, 아래처럼 말한다.
> 현재 문체 규칙은 잠정안입니다. 실제 샘플이 더 들어오면 더 정교하게 맞출 수 있습니다.

## 8. 스타일 적용 순서
1. 샘플 읽기
2. Style Card 생성
3. 이번 글 장르 확인
4. 장르 규칙과 충돌하는 습관 조정
5. 문체 강도 설정
6. 초안 또는 수정본에 반영
7. 스타일 드리프트 체크

## 9. 스타일 드리프트 체크
적용 후 아래를 본다.
- 사용자 같아 보이는가?
- 그러나 전달력은 좋아졌는가?
- 지나치게 흉내 내는 느낌은 없는가?
- 장르와 독자에 맞는가?
- 불필요한 상투어나 장식이 늘지 않았는가?

## 10. 깊은 문체 분석 요청이 들어오면
사용자가 문체 파일 자체를 만들고 싶어 하면 아래 순서를 따른다.
1. 분석 계획 제시
2. 사용자 승인
3. 샘플 정밀 분석
4. Style Card + 세부 규칙 + 금지 습관 정리
5. 재사용 가능한 텍스트/마크다운 가이드로 출력

## 11. 추천 출력 형식
### 간단 버전
- 이 문체의 핵심 5가지
- 적용 시 주의점 3가지
- 지금 글에 반영한 방식

### 상세 버전
- Style Card
- 장르별 보정 규칙
- Before/After 예시
- 금지 표현 목록

## 12. 기억할 점
문체는 **마지막 광택**이 아니라, 초안 단계부터 구조와 문장 리듬에 영향을 준다.
하지만 언제나 **목적, 독자, 장르 규칙**이 문체보다 위에 있다.

## 13. Style Card 내보내기/가져오기

### 내보내기 (세션 종료 시)
Style Card가 생성되었거나 변경된 세션의 끝에:
1. 최종 Style Card를 마크다운 코드블록으로 출력
2. "이 내용을 `style-card.md`로 저장하세요" 안내
3. 변경 이력 요약 (어떤 차원이 왜 바뀌었는지)

### 가져오기 (세션 시작 시)
사용자가 Style Card 파일을 업로드하면:
1. 파일 내용을 파싱하여 10차원 확인
2. "이전 Style Card를 불러왔습니다. 10차원 keep/avoid 기준으로 진행합니다."
3. 현재 장르와 충돌하는 습관이 있으면 조정 제안

### 피드백 축적 규칙 (autoresearch 패턴)

**피드백 분류**: 일회성 vs 지속적
- "이번만" / "여기만" → 일회성 (Style Card 미반영)
- "항상" / "앞으로" / "늘" → 지속적 (Style Card 반영)
- 모호할 때 → "이 피드백을 앞으로도 적용할까요?" 질문

**판정**: keep / discard
- keep: Style Card 해당 차원의 keep 또는 avoid 항목에 규칙 추가/수정
- discard: 현재 초안에만 적용, Style Card 미반영

사용자 피드백 유형별 Style Card 업데이트:
| 피드백 | 대응 차원 | 판정 | 업데이트 방법 |
|--------|---------|------|-------------|
| "더 짧게 써주세요" | 문장 길이, 문단 호흡 | keep | keep에 "짧은 문장" 추가 |
| "이건 내 스타일 아냐" | 전체 검토 | keep | 어떤 부분이 다른지 질문 → 해당 차원 수정 |
| "이 부분 좋아요" | 해당 차원 식별 | keep | 해당 keep 규칙 강화 |
| "더 전문적으로" | 어휘 수준, 목소리 | keep | keep에 "전문어 허용" 추가 |
| "친근하게" | 목소리, 관점 거리 | keep | keep에 "대화체/1인칭" 추가 |
| "이번만 격식체로" | 목소리 | discard | 현재 초안에만 적용 |

**변경 이력**: Style Card의 Change Log에 append
- 형식: `YYYY-MM-DD: keep/discard | dimension | reason | note`
